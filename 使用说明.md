# PRD Pilot 使用说明书

## 1. 项目定位

PRD Pilot 是一个面向产品经理的 AI PRD 与 Demo 快速生成工作台。

它不是单纯“帮你多生成几页内容”的工具，而是把一个模糊需求整理成一套可评审、可演示、可修改的方案。

当前主流程是：

`输入需求 -> 解析需求摘要 -> 生成 PRD / Demo / 原型说明 -> 一致性检查 -> 定点修改`

---

## 2. 启动方式

### 后端启动

```bash
cd prd-pilot/backend
pip install -r requirements.txt
python main.py
```

### 前端启动

```bash
cd prd-pilot/frontend
npm install
npm run dev
```

### 访问地址

- 前端：`http://localhost:5173`
- 后端健康检查：`http://localhost:8000/api/health`

## 3. 页面结构说明

页面分成三块：

### 右上角：模型配置按钮

点击页面右上角的 `配置模型` 按钮，会弹出模型配置窗口。

这里负责配置本次生成要使用的大模型。

字段说明：
- `模型平台`：例如 DeepSeek、OpenAI、OpenRouter、Moonshot、DashScope。
- 当前还支持：`Zhipu / GLM`、`Groq`、`Ollama (Local)`、`Custom OpenAI Compatible`。
- `模型名称`：可以从推荐列表里选，也可以手动输入兼容模型名。
- `API Key`：你自己的模型 Key。
- `Base URL`：兼容 OpenAI 的接口地址。
- `Max Tokens`：单次生成的输出上限，留空时默认按模型最大可用输出上限处理。

操作说明：
- 点 `测试连接` 可以验证当前模型配置是否可用。
- 点 `恢复默认` 可以回到后端默认配置。
- 这份配置只保存在当前浏览器，本地有效，不会写进服务器配置文件。

### 左侧中部：需求输入

这里负责填写原始需求。

有两种模式：

#### 快速模式

适合你只有一句想法的时候。

字段说明：
- `需求描述`：核心想法，必须填。
- `补充要求`：补充风格、展示重点、限制条件。
- `原型风格`：控制 Demo 的页面风格，例如后台管理台、移动端 H5。

#### 结构化模式

适合你已经有比较清晰的需求信息。

字段说明：
- `产品名称`：项目或功能名称。
- `产品类型`：例如 Web 工具、H5、后台系统。
- `需求描述`：要解决什么问题。
- `目标用户`：谁来用。
- `用户痛点`：用户当前最难受的点。
- `核心场景`：典型使用场景。
- `核心功能`：最重要的功能点。
- `页面偏好`：你希望出现哪些页面。
- `重点演示场景`：Demo 里优先展示的流程。
- `页面数量偏好`：例如 3 个核心页面。
- `界面偏好`：例如图表、表单、工作流。
- `原型风格`：Demo 风格参数。
- `约束条件`：例如不接真实后端。
- `补充说明`：其他你想强调的内容。

### 左侧下部：需求摘要 / Requirement Spec

这里是系统自动抽取后的结构化需求摘要。

它是后续所有生成、检查、修改的基线。也就是说：
- PRD 基于它生成
- Demo 基于它生成
- 一致性检查拿它做对照
- 定点修改也会先改它

如果系统抽取得不准，你可以直接在这里手动改。

### 右侧：评审产出

这里有 4 个标签页：

- `PRD`：查看生成的产品需求文档。
- `Demo`：查看可交互原型，可以切换预览 / HTML。
- `原型说明`：查看页面结构、操作路径、验证目标。
- `一致性检查`：查看 PRD、Demo、原型说明之间是否对齐。

### 底部：定点修改

这里不是“随便再生成一次”，而是告诉系统你要改哪里。

字段说明：
- `修改类型`：例如新增页面、调整目标用户、强化数据展示。
- `目标模块`：改 PRD、改 Demo，还是偏原型说明。
- `影响页面`：如果修改只针对某几个页面，就在这里选。
- `修改说明`：一句话说清楚你要怎么改。

## 4. 按钮说明

### 左侧主按钮

#### `解析需求摘要`
先把你的原始输入整理成 Requirement Spec。

建议：第一次使用时先点这个按钮。

#### `生成 PRD`
基于当前 Requirement Spec 生成 PRD。

如果你没有先点“解析需求摘要”，系统也会尽量先自动补出摘要再生成。

#### `生成 Demo 与原型说明`
基于当前 Requirement Spec 生成：
- Demo HTML
- 原型说明

#### `清空工作区`
清掉当前页面上的所有输入、摘要和生成结果。

### 右侧按钮

#### `测试连接`
验证当前模型平台、模型名、API Key、Base URL 是否可用。

#### `运行检查`
对当前 Requirement Spec、PRD、Demo、原型说明做一致性检查。

#### 复制按钮
复制当前标签页里的内容。

#### `下载 HTML`
下载当前 Demo 页面。

### 底部按钮

#### `更新 PRD`
根据“定点修改”里的内容，局部更新 PRD。

#### `更新 Demo`
根据“定点修改”里的内容，局部更新 Demo 和原型说明。

## 5. 推荐使用流程

### 场景一：你只有一句模糊想法

1. 在 `快速模式` 填写 `需求描述`
2. 点 `解析需求摘要`
3. 检查左下的 Requirement Spec 是否合理
4. 点 `生成 PRD`
5. 点 `生成 Demo 与原型说明`
6. 点 `运行检查`
7. 如果有问题，到底部 `定点修改`，再点 `更新 PRD` 或 `更新 Demo`

### 场景二：你已经有比较清晰的需求

1. 切换到 `结构化模式`
2. 把核心字段填完整
3. 点 `解析需求摘要`
4. 手动修一下 Requirement Spec
5. 继续生成 PRD 和 Demo
6. 最后做一致性检查和定点修改

---

## 6. 一致性检查怎么看

一致性检查会给你：

- `一致性等级`：high / medium / low
- `一致性评分`
- `检查结果`
- `问题列表`
- `修复建议`

重点关注这几类问题：
- 页面在 PRD 里写了，但 Demo 没做出来
- Demo 里有功能，但 PRD 没写
- 名称不统一
- 页面之间串不起来
- 原型说明和实际 Demo 对不上

如果你看到修复建议，可以直接点建议按钮，系统会把它带入到底部修改区。

---

## 7. 定点修改怎么用

### 示例 1：新增一个页面

- 修改类型：`新增一个页面`
- 目标模块：`Demo`
- 影响页面：选相关页面，或者不选
- 修改说明：`增加一个评审结果页，把关键问题和修复建议集中展示。`
- 点击：`更新 Demo`

### 示例 2：精简文档

- 修改类型：`精简 PRD 篇幅`
- 目标模块：`PRD`
- 修改说明：`删掉过长描述，保留背景、功能表、业务流程和风险项。`
- 点击：`更新 PRD`

### 示例 3：切换原型风格

- 修改类型：`切换原型风格`
- 目标模块：`Demo`
- 修改说明：`改成后台管理台风格，突出筛选区、列表区和状态标签。`
- 点击：`更新 Demo`

## 8. 常见问题

### 1. 为什么我点“生成 PRD”前建议先点“解析需求摘要”？

因为 Requirement Spec 是整个系统的输入基线。你先看一眼它，后面的 PRD 和 Demo 会更稳。

### 2. 为什么要先生成 PRD 再生成 Demo？

不是硬性要求，但推荐这样做。因为 Demo 会参考 PRD，生成结果通常更稳定。

### 3. 一致性检查分低怎么办？

先看“问题列表”和“修复建议”，然后用底部“定点修改”去修，不要直接整页重来。

### 4. Demo 下载下来后是什么？

是一个单文件 HTML，可以直接打开预览。

### 5. 页面生成后不满意怎么办？

不要直接清空重做。优先用：
- `运行检查`
- `定点修改`

这样结果会更可控。

---

## 9. 建议你实际这样用

最稳的用法是：

1. 先用快速模式写一句需求
2. 点 `解析需求摘要`
3. 手动修 Requirement Spec
4. 生成 PRD
5. 生成 Demo
6. 跑一致性检查
7. 根据修复建议做 1 到 2 轮定点修改

如果你只追求“一键全出”，结果通常会更飘。

这个版本最强的地方不是“一次生成很多内容”，而是：

- 先理解需求
- 再生成产物
- 再检查对齐
- 再局部修复
